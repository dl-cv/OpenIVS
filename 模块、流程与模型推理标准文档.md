# 模块、流程与模型推理标准文档

本文件记录 C# 与 C++ 两套 API 共用的模型、结果、Flow、模板与计时语义。语言特有的工程结构、依赖、固定加载路径与封装接口见各自 API 文档。

## 1. 系统总览

OpenIVS 的模型只有两类：

1. 普通模型，文件类型为 `.dvt`、`.dvo`，用于直接加载单个模型并完成推理。
2. 流程模型，文件类型为 `.dvst`、`.dvso`，用于把输入、预处理、模型推理、结果处理、可视化、模板匹配组织成一条流程。

无论调用端使用 C# 还是 C++，对外都围绕同一组业务概念工作：

1. 输入是一张图或一批图。
2. 输出既可以是结构化结果，也可以是 JSON 结果。
3. 结果中的类别、框、角度、区域、折线、模板匹配明细都使用同一套语义。
4. 推理结束后都可以读取本次请求的时间信息，用于界面显示、测试输出和性能定位。

### 1.1 对外对象

| 逻辑对象 | C# | C++ | 作用 | 直接产出 |
| --- | --- | --- | --- | --- |
| 普通模型 | `dlcv_infer_csharp.Model` | `dlcv_infer::Model` | 加载 `.dvt`、`.dvo` 模型并完成推理 | 结构化结果、JSON 结果、模型信息、时间信息 |
| 滑窗模型 | `dlcv_infer_csharp.SlidingWindowModel` | `dlcv_infer::SlidingWindowModel` | 把滑窗切图、子图推理、结果合并打包成一个模型对象 | 结构化结果、JSON 结果 |
| 流程模型 | `DlcvModules.FlowGraphModel` | `dlcv_infer::flow::FlowGraphModel` | 把一整条流程图当作模型对象加载和推理 | 结构化结果、JSON 结果、流程信息、时间信息 |
| DVS 流程归档模型 | `DlcvModules.DvsModel` | `dlcv_infer::flow::DvsModel` | 读取 `.dvst`、`.dvso` 归档并按流程模型执行 | 结构化结果、JSON 结果、归档内流程信息 |
| 工具类 | `dlcv_infer_csharp.Utils` | `dlcv_infer::Utils` | 提供模型释放、设备信息、GPU 信息、可视化转换等公共工具 | JSON 字符串、设备信息、性能辅助信息 |

### 1.2 命名方式

| 场景 | 命名方式 | 示例 |
| --- | --- | --- |
| JSON 字段 | `snake_case` | `category_id`、`with_mask`、`mask_rle` |
| C# 公共属性 | `PascalCase` | `CategoryId`、`WithMask`、`MaskRle` |
| C++ 公共成员 | `camelCase` | `categoryId`、`withMask`、`maskRle` |

### 1.3 一次请求的外部表现

从调用方视角看，一次推理请求始终经历同一条主线：

1. 传入图像或图像列表。
2. 入口对象根据模型文件类型完成普通模型或流程模型加载。
3. 图像在推理前被规整到模型可接受的位深和通道数。
4. 推理结果被组织为结构化对象。
5. 需要 JSON 时，结果被转换成固定语义的 JSON 数组。
6. 需要流程可视化、模板匹配、区域过滤、去重等能力时，在 Flow 模块中继续处理。
7. 本次请求的时间信息被记录下来，供界面、测试程序和日志读取。

## 2. 模型推理

### 2.1 `Model`

`Model` 是普通推理入口。它负责把“模型路径 + 设备 + 输入图像”组织成一次完整请求。对调用方来说，`Model` 需要同时解决四件事：

1. 模型怎么加载和释放。
2. 图像怎么规整到模型可接受的输入格式。
3. 推理结果怎样变成稳定的结构化对象和 JSON 结果。
4. 本次请求的时间信息怎样被记录和读取。

#### `Model` 对外能力

| 能力域 | 主要接口 | 对外行为 |
| --- | --- | --- |
| 生命周期 | 构造、`FreeModel()`、`Dispose()` / 析构 | 创建对象、加载模型、释放模型、回收资源 |
| 状态 | `modelIndex`、`OwnModelIndex`、模型类型识别状态 | 表示底层模型句柄、当前对象是否拥有释放权、当前加载的是哪类模型 |
| 模型信息 | `GetModelInfo()`、`GetCachedModelInfo()`、`GetCachedMaxShape()`、`GetMaxBatchSize()`、`GetResolvedSubModelBatchItems()` | 读取模型元数据、批量能力、输入形状信息和流程内模型信息 |
| 推理 | `Infer()`、`InferBatch()`、`InferOneOutJson()` | 单图推理、批量推理、单图 JSON 输出 |
| 时间读取 | C# 通过 `InferTiming`，C++ 通过 `Model` 静态接口读取 | 获取本次请求的模型推理时间、总时间、Flow 节点耗时 |

### 2.2 模型文件类型

对外使用时，模型文件只分为两类：

1. 普通模型文件：`.dvt`、`.dvo`。这类模型由 `Model` 直接加载，适合单模型推理。
2. 流程模型文件：`.dvst`、`.dvso`。这类模型由 `FlowGraphModel` 或 `DvsModel` 加载，适合把多步处理组织成一条完整流程。

调用端不需要为这两类模型准备两套完全不同的调用方式。传入模型路径、设备和请求参数后，入口对象会完成对应的加载与执行。

### 2.3 输入图像处理

模型推理前，输入图像会被规整到模型可接受的格式。调用端只关心“把图传进去就能得到稳定结果”，底层负责完成以下处理：

| 输入情况 | 处理方式 | 对调用方表现 |
| --- | --- | --- |
| 空图或空批量 | 直接报错 | 不返回伪成功结果 |
| 16 位、浮点、带符号整型图像 | 转为 8 位 | 模型得到稳定输入 |
| 模型期望 3 通道 | 统一按 RGB 语义推理 | 普通模型、流程模型都得到一致的颜色语义 |
| 模型期望 1 通道 | 统一转灰度 | 不要求调用方自己手动转灰度 |
| 批量输入 | 保持原顺序 | 返回结果与输入顺序一一对应 |
| Flow 前端注入图像 | 按 RGB 语义进入流程 | 前端与流程之间颜色语义固定 |
| 从磁盘读取的 OpenCV 图像 | 先按 BGR 读取，再在模型阶段转成模型需要的格式 | 读盘行为与模型行为解耦 |

### 2.4 模型信息

`GetModelInfo()` 返回的是调用端理解模型能力所需的根级信息。无论当前对象是普通模型还是流程模型，调用方都可以从中获得同一类信息：

| 信息项 | 含义 |
| --- | --- |
| `backend` | 当前对象属于普通模型还是流程模型 |
| `model_type` | 模型或流程对象的类型 |
| `input_shapes` | 输入形状信息，用于判断通道数、尺寸范围、批量能力 |
| `max_batch_size` | 模型一次可处理的最大批量 |
| `pipeline` | 流程模型或归档模型包含的流程内容 |
| `loaded_model_meta` | 流程里每个模型节点加载后的元信息 |

### 2.5 `SlidingWindowModel`

`SlidingWindowModel` 把“大图切小图、对子图推理、把子图结果合并回原图”作为一个完整模型对象来使用。调用方不需要自己管理切图和拼接。

| 参数 | 含义 |
| --- | --- |
| `model_path` | 底层检测模型路径 |
| `device_id` | 推理设备编号 |
| `small_img_width` | 滑窗宽度 |
| `small_img_height` | 滑窗高度 |
| `horizontal_overlap` | 横向重叠 |
| `vertical_overlap` | 纵向重叠 |
| `threshold` | 结果阈值 |
| `iou_threshold` | 合并时的 IoU 阈值 |
| `combine_ios_threshold` | 合并时的 IoS 阈值 |

## 3. 结果与几何信息

系统对外的结果围绕四类信息组织：

| 信息层 | 内容 | 用途 |
| --- | --- | --- |
| 识别信息 | 类别 ID、类别名、分数 | 用于业务判定、界面显示、文本输出 |
| 几何信息 | 框、角度 | 用于定位目标、绘制可视化、裁图、去重、区域过滤 |
| 区域信息 | `mask`、`mask_rle`、`poly` | 用于实例分割、区域着色、边界提取、轮廓分析 |
| 扩展信息 | `extra_info`、`metadata` | 用于模块附加信息、业务扩展信息、调试信息 |

### 3.1 结构化结果层级

| 类型 | 作用 | 结构 |
| --- | --- | --- |
| `ObjectResult` | 表示一个目标的完整结果 | 一个类别 + 一个几何位置 + 一个区域信息集合 |
| `SampleResult` | 表示一张图上的所有目标 | `results` 列表 |
| `Result` | 表示一次批量请求的所有图片结果 | `sample_results` 列表 |

### 3.2 `ObjectResult`

`ObjectResult` 是所有推理对象共享的基础结果对象。调用端拿到它以后，可以完成列表展示、框绘制、掩码绘制、文本输出、后处理和模板匹配。

| 字段 | 含义 | 典型用途 |
| --- | --- | --- |
| `category_id` | 类别编号 | 程序内逻辑判断、分类统计 |
| `category_name` | 类别名称 | 界面显示、文本输出、模板匹配 |
| `score` | 置信度 | 阈值过滤、日志输出、排序 |
| `area` | 面积 | 后处理过滤、结果统计 |
| `bbox` | 框位置 | 绘制框、裁剪、去重、区域判断 |
| `with_bbox` | 是否有框 | 区分有定位结果和无定位结果 |
| `with_angle` | 是否有角度 | 区分普通框和旋转框 |
| `angle` | 旋转角度 | 绘制旋转框、旋转裁剪、回正 |
| `with_mask` | 是否有 mask | 区分检测结果和分割结果 |
| `mask` | 结构化结果中的局部 mask 图像 | 做进一步图像计算 |
| `extra_info` | 扩展信息对象 | 放置折线、业务附加信息 |
| `metadata` | 元信息对象 | 放置流程来源、模块附加信息 |

### 3.3 JSON 结果

`InferOneOutJson()` 和 `FlowGraphModel` 的外部 JSON 输出都围绕固定语义组织。JSON 结果数组中的每一项都表示一条完整结果。

| JSON 字段 | 含义 | 说明 |
| --- | --- | --- |
| `category_id` | 类别编号 | 与结构化结果一致 |
| `category_name` | 类别名称 | 与结构化结果一致 |
| `score` | 置信度 | 与结构化结果一致 |
| `area` | 面积 | 与结构化结果一致 |
| `bbox` | 框位置 | 轴对齐框为 `[x, y, w, h]`，旋转框为 `[cx, cy, w, h]` |
| `with_bbox` | 是否有框 | 与结构化结果一致 |
| `with_angle` | 是否有角度 | 与结构化结果一致 |
| `angle` | 旋转角度 | 无角度时固定为 `-100` |
| `with_mask` | 是否有区域结果 | 与结构化结果一致 |
| `mask_rle` | RLE 编码区域 | 面向 JSON 传输和跨语言交换 |
| `poly` | 多边形轮廓数组 | 用于前端绘制、边界分析、折线提取 |
| `extra_info` | 扩展信息 | 例如 `extra_info.polyline` |
| `metadata` | 元信息 | 记录模块附加信息、运行信息 |

### 3.4 几何语义

| 内容 | 表达方式 | 坐标语义 |
| --- | --- | --- |
| 普通框 | `bbox=[x, y, w, h]` | 左上角与宽高 |
| 旋转框 | `bbox=[cx, cy, w, h]` + `angle` | 中心点、宽高、角度 |
| 没有角度 | `with_angle=false`，`angle=-100` | 固定语义，不靠缺字段判断 |
| `mask_rle` | RLE 编码后的区域 | 与结果对应区域一致 |
| `poly` | 轮廓点列表 | 原图坐标 |
| `extra_info.polyline` | 折线点列表 | 在进入最终 JSON 前还原到原图坐标 |

### 3.5 单图 JSON 结果示例

```json
[
  {
    "category_id": 1,
    "category_name": "demo",
    "score": 0.98,
    "area": 2400.0,

    "with_angle": false,
    "angle": -100.0,
    "with_mask": false,
    "mask_rle": null,
    "poly": null,
    "extra_info": {},
    "metadata": {}
  }
]
```

## 4. Flow 模型与流程执行

### 4.1 Flow 的外部意义

Flow 把一整条算法链路组织成一个可加载、可推理、可测速的对象。对界面、测试程序和业务代码来说，Flow 和普通模型一样，都可以被当作“传图进去，拿结果出来”的对象使用。

Flow 适合承载以下能力：

1. 固定输入方式。
2. 多步预处理。
3. 多模型串联。
4. 去重、过滤、覆盖、标签合并等后处理。
5. 可视化输出。
6. 模板生成、模板保存、模板加载、模板匹配。

### 4.2 `FlowGraphModel`

| 能力 | 说明 |
| --- | --- |
| `Load()` | 读取流程 JSON，并预加载所有 `model/*` 节点 |
| `GetModelInfo()` | 返回流程根对象及流程元信息 |
| `GetLoadedModelMeta()` | 返回流程中每个模型节点的加载信息 |
| `Infer()` / `InferBatch()` | 返回标准结构化结果 |
| `InferOneOutJson()` | 返回单图 JSON 结果 |
| `Benchmark()` | 返回平均耗时，单位毫秒 |

### 4.3 一次 Flow 请求的执行路径

1. `FlowGraphModel` 把单图或批量图像写入 `ExecutionContext`。
2. 前端注入图像以 RGB 语义进入流程。
3. `GraphExecutor` 读取节点列表，按 `order` 和 `id` 排序。
4. 模块按链路依次执行，图像、结果、模板和标量在模块之间流动。
5. `output/return_json` 把局部结果还原到原图坐标并聚合成对外结果。
6. `FlowGraphModel` 从聚合结果读取 `result_list`，同时返回时间信息。

### 4.4 `DvsModel`

`DvsModel` 负责把 `.dvst`、`.dvso` 归档文件变成可直接执行的流程对象。对调用方来说，它的外部行为与 `FlowGraphModel` 一样，区别只在于加载入口是一个归档文件。

#### 归档加载过程

1. 检查文件头是否为 `DV\n`。
2. 读取第二行 JSON 头。
3. 从头信息中读取 `file_list` 和 `file_size`。
4. 解包 `pipeline.json` 和归档中的其他文件。
5. 把流程中的 `model_path` 重写到临时目录里的真实文件路径。
6. 记录 `model_path_original` 和 `model_name`。
7. 调用流程加载逻辑完成模型预加载。
8. 清理解包产生的临时目录。

#### 加载报告

`Load()` 返回的结果围绕逐节点加载状态组织：

| 字段 | 含义 |
| --- | --- |
| `code` | 整体加载是否成功 |
| `message` | 整体加载结果文本 |
| `models` | 每个模型节点的加载结果列表 |
| `node_id` | 模型节点编号 |
| `type` | 模型节点类型 |
| `title` | 模型节点标题 |
| `model_path` | 当前加载路径 |
| `status_code` | 当前节点是否加载成功 |
| `status_message` | 当前节点加载结果文本 |

## 5. Flow 运行时对象

Flow 中的所有模块都围绕同一套运行时对象工作。这样做的直接效果是：图像、结果、模板、标量和上下文信息可以在任意模块之间稳定流转，C# 与 C++ 的行为也能够对应起来。

### 5.1 运行时对象

| 对象 | 作用 | 关键内容 |
| --- | --- | --- |
| `ExecutionContext` | 一次请求期间的共享上下文 | 前端图像、设备编号、推理参数、返回 JSON、时间累积等 |
| `TransformationState` | 记录原图到当前图的几何关系 | 原图尺寸、裁剪框、仿射矩阵、输出尺寸 |
| `ModuleImage` | 模块间传递图像的标准包装器 | 当前图、原图、变换状态、原图索引、滑窗元信息 |
| `ModuleChannel` | 描述一组模块输入或输出 | `ImageList`、`ResultList`、`TemplateList` |
| `ModuleIO` | 描述模块执行后的返回值 | `ImageList`、`ResultList`、`TemplateList` |
| `ModuleRegistry` | 根据 `type` 找到模块实现 | `moduleType -> Type` 或 `moduleType -> class` |
| `BaseModule` | 所有模块的共同基类 | 节点信息、属性、上下文、额外输入输出、标量输入输出 |
| `BaseInputModule` | 输入模块基类 | 忽略上游输入，通过 `Generate()` 产生首对输出 |
| `BaseModelModule` | 模型模块基类 | 管理内部 `Model` 对象，负责预加载与模型调用 |
| `GraphExecutor` | 流程执行器 | 节点排序、链路路由、标量注入、模块执行、时间记录 |

### 5.2 `ExecutionContext`

`ExecutionContext` 在一次请求里承担三类角色：

1. 承载入口输入，例如前端图像、图像列表、设备编号、推理参数。
2. 承载流程输出，例如 `frontend_json`、按节点组织的 JSON 结果、模型加载元信息。
3. 承载运行期累积值，例如模型推理时间累计值。

#### 常用上下文项

| 上下文项 | 作用 |
| --- | --- |
| `frontend_image_mat` | 单图入口图像 |
| `frontend_image_mats` | 批量入口图像 |
| `frontend_image_mat_list` | 批量入口图像列表 |
| `frontend_image_color_space` | 当前前端注入图像的颜色语义 |
| `frontend_image_path` | 前端图像路径 |
| `device_id` | 当前请求使用的设备编号 |
| `infer_params` | 推理参数对象 |
| `return_json_emit_poly` | 是否把轮廓写入最终 JSON |
| `loaded_model_meta` | 流程中每个模型节点的加载元信息 |
| `frontend_json` | 当前流程的最终 JSON 结果 |
| `frontend_json_by_node` | 按节点组织的 JSON 结果 |

### 5.3 `TransformationState`

`TransformationState` 用来回答一个核心问题：当前模块看到的结果，怎样回到原图坐标系。

| 内容 | 含义 |
| --- | --- |
| `original_width` / `original_height` | 原图尺寸 |
| `crop_box` | 从原图或父图裁出的区域 |
| `affine_2x3` | 原图到当前图的仿射变换 |
| `output_size` | 当前图尺寸 |

当流程里发生裁图、缩放、旋转、滑窗切图等操作时，结果不会直接丢失和原图的关系，后续 `output/return_json` 和可视化模块可以依靠 `TransformationState` 把结果还原回原图坐标。

### 5.4 `GraphExecutor`

`GraphExecutor` 负责把流程图上的节点关系变成实际执行过程。

| 行为 | 表现 |
| --- | --- |
| 节点排序 | 按 `order` 升序执行；`order` 相同时按 `id` 升序执行 |
| 链路路由 | 预先建立 `linkId -> (srcNodeId, srcOutIdx)` 映射 |
| 主通道与额外通道 | 每两个输入端口形成一组通道；第 0 组为主通道，其余进入 `ExtraInputsIn` |
| 标量注入 | 把上游标量结果写入 `ScalarInputsByIndex` 和 `ScalarInputsByName` |
| 模型预加载 | `LoadModels()` 只对 `model/*` 节点调用 `LoadModel()` |
| 输出发布 | 每个节点执行后，把主输出写入公共输出表，把额外输出写入 `ExtraOutputs` |
| 时间记录 | 记录图搭建、节点执行、输出发布等耗时 |

### 5.5 标量端口

标量端口用于在模块之间传递开关、数字和字符串，不走图像通道和结果通道。

| 端口类型字符串 | 语义 |
| --- | --- |
| `bool`、`boolean` | 布尔值 |
| `int`、`integer` | 整数 |
| `string`、`str`、`scalar` | 字符串 |

### 5.6 `output/return_json`

`output/return_json` 是 Flow 对外结果的最终聚合节点。调用方真正看到的流程 JSON 结果来自这里，而不是来自某个局部模块。

它负责完成以下工作：

1. 把局部坐标结果还原到原图坐标。
2. 以 `origin_index` 为依据，把批量输入中的结果按原图归组。
3. 把结果写入 `frontend_json.last` 与 `frontend_json.by_node`。
4. 供 `FlowGraphModel` 读取并组织成 `result_list`。
5. 对同一节点、同一张图、同一次请求内的重复结果做去重。

## 7. 模板与模板匹配

模板能力服务于同一类业务：把某个产品、某个相机位、某套 OCR 结果保存成可复用模板，再把后续识别结果与模板进行比对，并输出清楚的匹配明细。

模板功能围绕四个动作工作：

1. 从 OCR 结果生成模板。
2. 把模板保存到文件。
3. 从文件读取模板。
4. 把当前识别结果与模板做逐项比对。

### 7.1 模板链路中的对象关系

| 对象 | 作用 |
| --- | --- |
| `SimpleOcrItem` | 表示一条 OCR 项，包括文本、位置、尺寸、置信度和匹配状态 |
| `SimpleTemplate` | 表示一个完整模板，包括模板标识、产品标识、相机位、OCR 项列表和匹配阈值 |
| `SimpleTemplateMatchDetail` | 表示一次模板匹配的明细结果，包括通过项、偏差项、过检项、漏检项和误判配对 |

### 7.2 `SimpleOcrItem`

`SimpleOcrItem` 表示模板中的一条 OCR 项，也表示当前识别结果中的一条 OCR 项。模板匹配时，系统按它的文本、位置、尺寸和置信度进行比较。

| 内容 | 对应属性 | 说明 |
| --- | --- | --- |
| 识别文本 | `Text` / `text` | 当前 OCR 项的文本内容 |
| 置信度 | `Confidence` / `confidence` | 当前 OCR 项的识别分数 |
| 左上角 X | `X` / `x` | OCR 项位置 |
| 左上角 Y | `Y` / `y` | OCR 项位置 |
| 宽度 | `Width` / `width` | OCR 项宽度 |
| 高度 | `Height` / `height` | OCR 项高度 |
| 识别时间 | `RecognitionTime` / `recognition_time` | 该 OCR 项产生的时间 |
| 匹配状态 | `MatchStatus` / `match_status` | 该 OCR 项在模板匹配中的状态 |

### 7.3 `SimpleTemplate`

`SimpleTemplate` 表示一个完整模板。它不只保存 OCR 文本，还保存产品归属、相机位和位置容差、尺寸容差、文本匹配方式等规则。

| 内容 | 对应属性 | 说明 |
| --- | --- | --- |
| 模板编号 | `TemplateId` / `template_id` | 模板唯一标识 |
| 模板名称 | `TemplateName` / `template_name` | 界面和文件中显示的模板名称 |
| 产品编号 | `ProductId` / `product_id` | 当前模板属于哪个产品 |
| 产品名称 | `ProductName` / `product_name` | 当前模板所属产品名称 |
| OCR 项列表 | `OCRResults` | 模板内保存的 OCR 项集合 |
| 相机位 | `CameraPosition` / `camera_position` | 模板属于哪个相机位或面位 |
| 期望位置 X | `ExpectedX` / `expected_x` | 模板的整体参考位置 |
| 期望位置 Y | `ExpectedY` / `expected_y` | 模板的整体参考位置 |
| 期望宽度 | `ExpectedWidth` / `expected_width` | 模板整体参考宽度 |
| 期望高度 | `ExpectedHeight` / `expected_height` | 模板整体参考高度 |
| 期望文本 | `ExpectedText` / `expected_text` | 整体期望文本或关键文本 |
| 位置容差 | `PositionTolerance` / `position_tolerance` | OCR 项位置可偏移范围 |
| 尺寸容差 | `SizeTolerance` / `size_tolerance` | OCR 项尺寸可偏移范围 |
| 最低置信度 | `MinConfidence` / `min_confidence` | 模板匹配时允许的最低分数 |
| 是否严格文本匹配 | `StrictTextMatch` / `strict_text_match` | 文本比较是严格一致还是允许宽松处理 |
| 创建时间 | `CreatedTime` / `created_time` | 模板创建时间 |
| 更新时间 | `UpdatedTime` / `updated_time` | 模板最后更新时间 |
| 启用状态 | `IsEnabled` / `is_enabled` | 当前模板是否参与匹配 |
| 模板图路径 | `ImagePath` / `image_path` | 模板关联的图片路径 |

### 7.4 `SimpleTemplateMatchDetail`

`SimpleTemplateMatchDetail` 表示一次模板匹配的完整说明。它不仅给出“通过或失败”，还要说明错在哪里。

| 内容 | 对应属性 | 说明 |
| --- | --- | --- |
| 是否匹配成功 | `IsMatch` | 当前样本是否通过模板匹配 |
| 匹配分数 | `Score` | 本次匹配的综合得分 |
| 原因列表 | `Reasons` | 本次匹配成功或失败的文字说明 |
| 正确项 | `CorrectItems` / `correct_items` | 文本和位置都满足要求的项 |
| 偏差项 | `DeviationItems` / `deviation_items` | 文本、位置、尺寸或置信度存在偏差的项 |
| 过检项 | `OverDetectionItems` / `over_detection_items` | 当前识别出来但模板中不存在的项 |
| 漏检项 | `MissedTemplateItems` / `missed_template_items` | 模板中存在但当前没有识别到的项 |
| 误判配对 | `MisjudgmentPairs` / `misjudgment_pairs` | 当前识别项和模板项被错误配对的明细 |

### 7.5 模板匹配结果的阅读方式

模板匹配输出不是单一布尔值，而是一份可读的业务说明：

1. `IsMatch` 用于直接给出通过或失败。
2. `Score` 用于表达整体接近程度。
3. `Reasons` 用于向界面、日志、测试程序解释结果。
4. `CorrectItems` 表示命中的 OCR 项。
5. `DeviationItems` 表示命中了但存在偏差。
6. `OverDetectionItems` 表示出现了模板外的额外识别项。
7. `MissedTemplateItems` 表示模板里应出现但本次没识别到的项。
8. `MisjudgmentPairs` 表示模板项和识别项被配错的情况。

## 8. 计时与诊断

计时信息是运行时行为的一部分，不是附加说明。界面、测试程序和性能排查都依赖同一套时间数据。一次请求需要同时回答三个问题：

1. 本次请求总共用了多久。
2. 其中真正花在底层模型推理上的时间有多少。
3. 如果是流程推理，慢在流程的哪个节点。

### 8.1 统计项

| 统计项 | 对外名称 | 说明 | 用途 |
| --- | --- | --- | --- |
| 模型推理时间 | `dlcv_infer_ms` | 当前请求中所有底层模型推理调用消耗的时间总和 | 区分算法推理成本和流程调度成本 |
| 流程总时间 | `flow_infer_ms` | 从流程入口开始到最终结果组织完成的总时间 | 显示用户感知的整次请求耗时 |
| 节点耗时列表 | `node_timings` | 每个节点或流程阶段的耗时 | 定位慢模块、慢阶段 |

### 8.2 节点耗时项

`node_timings` 中每一项都用于说明某个节点或某个流程阶段花了多少时间。

| 字段 | 含义 |
| --- | --- |
| `node_id` | 节点编号 |
| `node_type` | 节点类型 |
| `node_title` | 节点标题 |
| `elapsed_ms` | 当前节点或阶段耗时，单位毫秒 |

### 8.3 普通模型与流程模型的时间表现

| 场景 | 模型推理时间 | 流程总时间 | 节点耗时 |
| --- | --- | --- | --- |
| 普通模型请求 | 直接等于本次请求的模型推理耗时 | 与整次请求耗时一致 | 为空或不使用 |
| Flow 请求 | 累加流程里所有底层模型调用时间 | 覆盖整个流程执行与结果组织过程 | 记录图搭建、节点执行、结果发布等时间 |

### 8.4 时间数据的读取入口

| 语言 | 读取入口 | 读取结果 |
| --- | --- | --- |
| C# | `DlcvModules.InferTiming.GetLast(...)`、`DlcvModules.InferTiming.GetLastFlowNodeTimings()` | 模型推理时间、总时间、节点耗时列表 |
| C++ | `dlcv_infer::Model::GetLastInferTiming(...)`、`dlcv_infer::Model::GetLastFlowNodeTimings()` | 模型推理时间、总时间、节点耗时列表 |

### 8.5 时间信息在产品侧的用途

时间信息直接服务于四类场景。界面需要展示本次请求耗时和每个模块耗时；控制台测试需要输出模型耗时、总耗时和节点明细；性能定位需要判断慢点在底层模型、流程组织还是具体模块；模型对比需要比较不同模型、不同流程和不同设备下的耗时表现。
