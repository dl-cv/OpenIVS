# C#测试程序2开发文档（双模型+滑窗+角度归一）

> 目标：任何开发者只看本文档，即可复刻一个功能与用户体验一致的 `C# 测试程序2`。允许代码组织不同，但主界面、操作流程、结果文本、可视化行为和核心业务规则必须一致。

## 1. 文档范围与定位

- 软件名称：`C# 测试程序2`
- 项目名称：`DlcvDemo2`
- 解决方案：`OpenIVS.sln`
- 技术栈：`.NET Framework 4.7.2`、`WinForms`、`OpenCvSharp`、`Newtonsoft.Json`、`DlcvCsharpApi`、`ImageViewer`
- 功能定位：面向固定业务场景的双模型推理 Demo，直接执行“模型1滑窗检测 -> 全图合并 -> ROI角度归一 -> 模型2精检测 -> 主器件改名 -> 坐标回写 -> 可视化展示”

本程序不是通用流程编辑器，也不是 SDK 全能力展示页。界面和逻辑都围绕固定 pipeline 展开，目标是让用户能够稳定完成一条业务闭环，而不是自由拼装任意推理流程。

## 2. 运行环境与依赖

- 操作系统：`Windows 10/11 x64`
- 编译平台：`x64`
- 启动项目：`DlcvDemo2`
- 主窗体类型：`WinForms Form`
- 图像显示控件：`DLCV.ImageViewer`
- 依赖项目：`DlcvCsharpApi`、`ImageViewer`

### 2.1 模型加载约束

- 模型1和模型2都通过 `new Model(path, 0, false)` 加载。
- `device_id` 固定为 `0`，界面没有 GPU/CPU 选择控件。
- 程序不提供 RPC 模式开关。
- 现场机器若没有可用的 `GPU 0`，模型加载会直接报错，文本区显示错误信息，不自动回退 CPU。

### 2.2 模型文件类型

- 支持后缀：`*.dvt`、`*.dvo`、`*.dvp`、`*.dvst`、`*.dvso`、`*.dvsp`
- 模型1和模型2均使用同一套文件过滤器。

## 3. 功能边界

### 3.1 必须具备的功能

- 分别选择并自动加载模型1、模型2（选择文件即加载）

- 在文本区输出结果摘要和测速统计
- 在图像区显示最终回写到整图坐标后的可视化结果
- 记忆最近一次模型1路径、模型2路径、图片路径和滑窗参数

### 3.2 明确不提供的功能

- 不提供 GPU 选择控件

## 5. 交互流程

### 5.1 加载模型1

- 点击 `模型1路径` 右侧 `浏览...`，选择模型1文件。
- 选择完成后立即实例化模型1。
- 加载成功后，文本区输出模型1加载成功信息和模型路径。
- 加载失败时，文本区显示异常信息。
- 新路径会写入 `LastModel1Path`。

### 5.2 加载模型2

- 点击 `模型2路径` 右侧 `浏览...`，选择模型2文件。
- 选择完成后立即实例化模型2。
- 加载成功后，文本区输出模型2加载成功信息和模型路径。
- 加载失败时，文本区显示异常信息。
- 新路径会写入 `LastModel2Path`。

### 5.3 选择图片

- 点击 `图片路径` 右侧 `浏览...`，选择待推理图片。
- 图片路径写入 `LastImagePath`。
- 选择完成后自动触发一次推理，无需手动点击 `执行推理`。
- `执行推理` 按钮仍保留，支持用户在不重新选图的情况下再次推理。

### 5.4 执行推理

- 推理触发方式有两种：选择图片后自动触发、点击 `执行推理` 手动触发。
- 触发后，程序先校验模型1、模型2和图片路径是否就绪。
- 任一项缺失时，文本区给出明确提示，不进入推理。
- 推理逻辑在后台线程执行（例如 `Task.Run` / `BackgroundWorker`），避免阻塞 UI 线程。
- 进度条始终显示：空闲状态为 `0%`，推理中按阶段递增，完成后显示 `100%`，下一次推理开始前重置为 `0%`。
- 推理过程中通过 UI 线程安全回调更新进度与状态文本，至少覆盖以下阶段：
  - 图片解码与参数检查
  - 模型1滑窗推理
  - 全图合并与 ROI 准备
  - 模型2逐 ROI 推理与回写
  - 可视化刷新与文本汇总
- 推理进行中禁止重复进入同一推理流程（再次选图或点击按钮时给出提示）。
- 推理成功后：
  - 图像区显示整图与最终检测结果。
  - 文本区输出本次推理摘要。
  - 当前图像保留在界面中，方便重复测速或调参后再次执行。

### 5.5 执行测速

- 点击 `测速` 后，对当前图片重复执行完整双模型 pipeline。
- `测速轮数` 控制重复次数。
- 测速与单次推理互斥；若当前正在推理，测速请求不进入执行。
- 测速结果只更新文本区，不覆盖图像区上一轮正常推理的可视化结果。
- 测速统计包含总耗时、平均耗时、最快耗时、最慢耗时。

### 5.6 释放模型

- 点击 `释放模型` 后，释放模型1和模型2实例。
- 已记忆的模型路径、图片路径和滑窗参数不清空。
- 图像区保持当前显示内容，文本区输出释放完成信息。

## 6. 固定推理流程

### 6.1 第一步：整图滑窗与模型1推理

- 按当前界面的滑窗参数对整图切片。
- 默认参数为：
  - `窗口宽 = 2560`
  - `窗口高 = 2560`
  - `水平重叠 = 1024`
  - `垂直重叠 = 1024`
- 对应步长为：
  - `step_x = 1536`
  - `step_y = 1536`
- 边缘窗口采用回退策略，保证最右侧和最下侧也被完整覆盖。
- 每个窗口都保留其在整图中的偏移量。
- 模型1对每个窗口单独推理，得到窗口内检测结果。
- 每条结果在窗口阶段就被抬升到整图坐标系。

### 6.2 第二步：模型1全图级合并

- 所有窗口结果进入统一的全图级去重合并。
- 合并粒度按完整 `category_name` 进行，不按基础器件名降维。
- `电阻0` 与 `电阻90` 不会被错误合并。
- 合并阈值固定为：
  - `iou_threshold = 0.2`
  - `combine_ios_threshold = 0.2`
- 这里使用窗体内的全图几何合并逻辑，不直接复用 `DlcvCsharpApi/SlidingMerge.cs` 当前实现。

### 6.3 第三步：解析类别角度并裁剪 ROI

合并后的每个模型1目标都会进入逐目标处理。

- 模型1的 `category_name` 包含基础类别和角度后缀。
- 解析规则使用：
  - 正则：`^(.*?)(0|90|180|270)$`
- 解析成功时：
  - `base_name` 为前缀类别名
  - `angle` 为结尾角度
- 解析失败时：
  - `base_name = 原始 category_name`
  - `angle = 0`

ROI 裁剪规则如下：

- 模型1若输出普通框，则按 `xywh` 从原图裁剪。
- 模型1若输出旋转框或带角度信息，则按旋转裁剪逻辑从原图获取 ROI。
- ROI 无效时跳过当前目标，并在文本区记录该目标被跳过。

### 6.4 第四步：ROI 逆时针归一

- ROI 按模型1解析出的角度逆时针旋转回 `0` 度视角。
- 例如模型1类别为 `电阻90`，则该 ROI 逆时针旋转 `90` 度后送入模型2。
- 角度归一仅作用于模型2输入，不直接修改最终回写坐标。

### 6.5 第五步：模型2局部推理

- 模型2接收归一后的 ROI。
- 模型2输出局部图中的多目标检测结果。
- 模型2常见类别包含：
  - 主器件本体
  - 字符
  - 引脚
  - 焊点
  - 其他细节类

### 6.6 第六步：主器件选择与类别替换

模型2输出后执行固定业务规则：

1. 先排除 `字符`、`引脚`、`焊点`。
2. 对剩余候选按以下优先级排序：
   - 面积最大优先
   - 面积相同则 `score` 更高优先
   - 再相同则保留原始顺序更靠前者
3. 选中的结果作为主器件。
4. 主器件的 `category_name` 替换为模型1原始类别名。
5. 其他模型2结果全部保留，不删除、不改名。

这一步的核心效果是：

- 模型1提供“器件大类 + 角度”
- 模型2提供“局部精检测”
- 最终主器件名称保留模型1的方向信息，例如 `电阻90`

### 6.7 第七步：模型2结果回写到整图

- 模型2坐标最初位于“旋转归一后的 ROI 坐标系”。
- 回写顺序固定为：
  1. 从归一坐标系逆旋转回 ROI 原始方向
  2. 再叠加 ROI 在整图中的偏移量
- 回写完成后，所有结果统一落在整图坐标系中。
- 图像区绘制和文本区输出都只使用整图坐标结果。

## 7. 结果组织与界面展示

### 7.1 最终结果结构

最终结果组织为单一整图结果列表，用于直接驱动 `imagePanel1.UpdateImageAndResult(...)`。

每条结果至少包含：

- `category_name`
- `score`
- `bbox` 或 `rotated_bbox`
- 整图坐标

### 7.2 图像区展示规则

- 图像区始终显示原图。
- 可视化对象只使用最终结果，不展示中间滑窗框和中间 ROI。
- 每条结果都绘制边界框和标签。
- 标签格式为：`{category_name} {score:F2}`
- 用户可以在图像区进行滚轮缩放、左键拖拽和右键复位。
- 无结果时仍显示原图，文本区输出“未检测到结果”。

### 7.3 文本区展示规则

文本区显示以下信息：

- 当前图片路径
- 当前模型1路径
- 当前模型2路径
- 当前滑窗参数
- 滑窗数量
- 模型1合并后目标数
- 最终结果数
- 结果明细
- 测速统计
- 推理阶段与进度状态
- 错误与异常信息

## 8. 关键业务规则示例

### 8.1 主规则示例

模型1某条合并结果为：

- `category_name = 电阻90`

对应 ROI 送入模型2后得到：

- `电阻`
- `电阻`
- `焊点`

最终处理结果为：

- 面积最大的 `电阻` 被认定为主器件
- 该主器件名称被替换为 `电阻90`
- 另一个 `电阻` 保留原类别
- `焊点` 保留原类别
- 上述全部结果最终都回写到整图坐标中

### 8.2 边界规则示例

- 模型2无结果：保留模型1兜底结果，类别名保持模型1原值
- 模型2只有 `字符/引脚/焊点`：这些结果全部保留，但不强行改名
- 模型1类别角度解析失败：按 `0` 度处理
- 滑窗重叠区域出现重复目标：先映射到整图，再按全图规则合并为一条

## 9. 异常与兜底行为

- 角度解析失败：流程继续执行，当前目标按 `angle = 0` 处理
- ROI 无效：跳过当前目标并记录日志，不中断整图流程
- 模型2无结果：保留模型1兜底结果
- 模型2只有被排除类：保留这些结果，不做主器件改名
- 模型加载失败：文本区显示异常信息，界面保持可继续操作状态
- 图片解码失败：文本区显示失败信息，不崩溃
- 推理后台线程抛异常：异常回传主线程并显示在文本区，进度条与推理触发控件状态正确复位
- 推理进行中再次触发：提示“当前正在推理”，忽略重复请求
- 释放模型失败：文本区提示异常，但界面不中断

## 10. 验收清单

- 窗口标题、布局和控件文本与本文档一致
- 模型1路径、模型2路径、图片路径支持记忆恢复
- 滑窗参数支持记忆恢复
- 界面中不包含 `加载模型1`、`加载模型2` 按钮
- 选择模型1/模型2文件后会自动加载
- 选择图片后会自动触发推理，`执行推理` 按钮仍可手动触发
- 推理进度条默认显示 `0%` 且始终可见，推理阶段状态可见
- 推理在后台线程执行，窗口拖拽/最小化/切换时 UI 不明显卡死
- 模型加载固定使用 `device_id = 0`
- 界面中没有 GPU 选择、JSON 输出和一致性测试入口
- 整图滑窗参数默认为 `2560/2560/1024/1024`
- 滑窗边缘覆盖完整，不漏右边界和下边界
- 模型1结果先抬升整图坐标，再做全图级去重合并
- 合并按完整 `category_name` 生效，不混淆 `电阻0` 与 `电阻90`
- 模型2主器件选择时严格排除 `字符`、`引脚`、`焊点`
- 主器件名称会被替换为模型1带角度类别名
- 其他模型2结果全部保留
- 最终结果全部以整图坐标显示
- 文本区和图像区能互相印证结果正确性
- 测速结果稳定可读，且不覆盖正常推理图像显示
