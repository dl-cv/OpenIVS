# C#测试程序2开发文档（三模型+滑窗+角度归一+分类分流）

> 目标：任何开发者只看本文档，即可复刻一个功能与用户体验一致的 `C# 测试程序2`。允许代码组织不同，但主界面、操作流程、结果文本、可视化行为和核心业务规则必须一致。

## 1. 文档范围与定位

- 软件名称：`C# 测试程序2`
- 项目名称：`DlcvDemo2`
- 解决方案：`OpenIVS.sln`
- 技术栈：`.NET Framework 4.7.2`、`WinForms`、`OpenCvSharp`、`Newtonsoft.Json`、`DlcvCsharpApi`、`ImageViewer`
- 功能定位：面向固定业务场景的三模型推理 Demo，直接执行“元件提取模型整板滑窗检测 -> 全图合并 -> 元件外扩 -> 类别名拆分与角度解析 -> ROI 逆时针归一 -> 按类别分流到元件检测模型 / IC 检测模型 -> 坐标回写 -> 最终结果合并展示”

本程序不是通用流程编辑器，也不是 SDK 全能力展示页。界面和逻辑都围绕固定 pipeline 展开，目标是让用户能够稳定完成一条业务闭环，而不是自由拼装任意推理流程。

## 2. 运行环境与依赖

- 操作系统：`Windows 10/11 x64`
- 编译平台：`x64`
- 启动项目：`DlcvDemo2`
- 主窗体类型：`WinForms Form`
- 图像显示控件：`DLCV.ImageViewer`
- 依赖项目：`DlcvCsharpApi`、`ImageViewer`

### 2.1 模型加载约束

- 元件提取模型、元件检测模型、IC 检测模型都通过 `new Model(path, 0, false)` 加载。
- `device_id` 固定为 `0`，界面没有 GPU/CPU 选择控件。
- 程序不提供 RPC 模式开关。
- 现场机器若没有可用的 `GPU 0`，模型加载会直接报错，文本区显示错误信息，不自动回退 CPU。

### 2.2 模型文件类型

- 支持后缀：`*.dvt`、`*.dvo`、`*.dvp`、`*.dvst`、`*.dvso`、`*.dvsp`
- 三个模型均使用同一套文件过滤器。

## 5. 交互流程

### 5.1 加载元件提取模型

- 点击 `元件提取模型路径` 右侧 `浏览...`，选择模型文件。
- 选择完成后立即实例化元件提取模型。
- 加载成功后，文本区输出加载成功信息和模型路径。
- 加载失败时，文本区显示异常信息。
- 新路径会写入 `LastExtractModelPath`。

### 5.2 加载元件检测模型

- 点击 `元件检测模型路径` 右侧 `浏览...`，选择模型文件。
- 选择完成后立即实例化元件检测模型。
- 加载成功后，文本区输出加载成功信息和模型路径。
- 加载失败时，文本区显示异常信息。
- 新路径会写入 `LastComponentDetectModelPath`。

### 5.3 加载 IC 检测模型

- 点击 `IC检测模型路径` 右侧 `浏览...`，选择模型文件。
- 选择完成后立即实例化 IC 检测模型。
- 加载成功后，文本区输出加载成功信息和模型路径。
- 加载失败时，文本区显示异常信息。
- 新路径会写入 `LastIcDetectModelPath`。

### 5.4 选择图片

- 点击 `图片路径` 右侧 `浏览...`，选择待推理图片。
- 图片路径写入 `LastImagePath`。
- 选择完成后自动触发一次推理，无需手动点击 `执行推理`。
- `执行推理` 按钮仍保留，支持用户在不重新选图的情况下再次推理。

### 5.5 执行推理

- 推理触发方式有两种：选择图片后自动触发、点击 `执行推理` 手动触发。
- 触发后，程序先校验元件提取模型、元件检测模型、IC 检测模型和图片路径是否就绪。
- 任一项缺失时，文本区给出明确提示，不进入推理。
- 推理逻辑在后台线程执行（例如 `Task.Run` / `BackgroundWorker`），避免阻塞 UI 线程。
- 进度条始终显示：空闲状态为 `0%`，推理中按阶段递增，完成后显示 `100%`，下一次推理开始前重置为 `0%`。
- 推理过程中通过 UI 线程安全回调更新进度与状态文本，至少覆盖以下阶段：
  - 图片解码与参数检查
  - 元件提取模型滑窗推理
  - 全图合并与 ROI 准备
  - 元件检测模型 / IC 检测模型局部推理与回写
  - 可视化刷新与文本汇总
- 推理进行中禁止重复进入同一推理流程（再次选图或点击按钮时给出提示）。
- 推理成功后：
  - 图像区显示整图与最终检测结果。
  - 文本区输出本次推理摘要。
  - 当前图像保留在界面中，方便调参后再次执行。

### 5.6 释放模型

- 点击 `释放模型` 后，释放元件提取模型、元件检测模型和 IC 检测模型实例。
- 已记忆的模型路径、图片路径和滑窗参数不清空。
- 图像区保持当前显示内容，文本区输出释放完成信息。

## 6. 固定推理流程

### 6.1 第一步：整图滑窗与元件提取模型推理

- 按当前界面的滑窗参数对整板图像切片。
- 默认参数为：
  - `窗口宽 = 2560`
  - `窗口高 = 2560`
  - `水平重叠 = 1024`
  - `垂直重叠 = 1024`
- 对应步长为：
  - `step_x = 1536`
  - `step_y = 1536`
- 边缘窗口采用回退策略，保证最右侧和最下侧也被完整覆盖。
- 每个窗口都保留其在整图中的偏移量。
- 元件提取模型对每个窗口单独推理，得到窗口内元件结果。
- 每条结果在窗口阶段就被抬升到整图坐标系。

### 6.2 第二步：元件提取结果全图级合并

- 所有窗口结果进入统一的全图级去重合并。
- 合并粒度按完整 `category_name` 进行，不按基础器件名降维。
- `IC0` 与 `IC90`、`电阻0` 与 `电阻270` 不会被错误合并。
- 合并阈值固定为：
  - `iou_threshold = 0.2`
  - `combine_ios_threshold = 0.2`
- 这里使用窗体内的全图几何合并逻辑，不直接复用 `DlcvCsharpApi/SlidingMerge.cs` 当前实现。

### 6.3 第三步：元件外扩、类别拆分与 ROI 裁剪

合并后的每个元件提取目标都会进入逐目标处理。

- 先基于当前目标框执行 `元件外扩`，默认值为 `32`。
- 外扩规则为在 ROI 四周各扩 `32` 像素，再裁剪到原图有效边界内。
- 元件提取模型的 `category_name` 包含“基础类别 + 角度后缀”。
- 解析规则使用：
  - 正则：`^(.*?)(0|90|180|270)$`
- 解析成功时：
  - `base_name` 为前缀类别名
  - `angle` 为结尾角度
- 解析失败时：
  - `base_name = 原始 category_name`
  - `angle = 0`
- 典型示例：
  - `IC90 -> base_name=IC, angle=90`
  - `电阻270 -> base_name=电阻, angle=270`

ROI 裁剪规则如下：

- 元件提取模型若输出普通框，则按外扩后的 `xywh` 从原图裁剪。
- 元件提取模型若输出旋转框或带角度信息，则按外扩后的旋转裁剪逻辑从原图获取 ROI。
- ROI 无效时跳过当前目标，并在文本区记录该目标被跳过。

### 6.4 第四步：ROI 逆时针归一到 0 度

- ROI 按解析出的角度逆时针旋转回 `0` 度视角。
- 例如元件提取模型类别为 `IC90`，则该 ROI 逆时针旋转 `90` 度后再进入后续检测。
- 再例如类别为 `电阻270`，则 ROI 逆时针旋转 `270` 度后进入后续检测。
- 角度归一仅作用于局部检测输入，不直接修改最终回写坐标。

### 6.5 第五步：按类别分流到对应检测模型

- 分流依据是拆分后的 `base_name`，不是原始带角度的 `category_name`。
- 当 `base_name` 为 `IC` 或 `BGA` 时，当前 ROI 路由到 `IC 检测模型`。
- 其余类别统一路由到 `元件检测模型`。
- 分流决策在每个 ROI 上独立进行，同一张整板图中可以同时走两条分支。

### 6.6 第六步：对应检测模型执行局部推理

- 被分配到的检测模型接收“已外扩、已归一到 0 度”的 ROI。
- 元件检测模型输出普通元件区域中的局部检测结果。
- IC 检测模型输出 IC / BGA 相关区域中的局部检测结果。
- 两个检测模型都允许输出多目标结果。
- 最终类别名以对应检测模型的输出为准，不再执行“主器件改名”逻辑。
- 若当前 ROI 对应的检测模型无结果，可保留该条元件提取结果作为兜底结果，避免目标整体丢失。

### 6.7 第七步：局部结果回写到整图并统一合并展示

- 局部检测结果最初位于“旋转归一后的 ROI 坐标系”。
- 回写顺序固定为：
  1. 从归一坐标系逆旋转回 ROI 原始方向
  2. 再叠加 ROI 在整图中的偏移量
- 回写完成后，元件检测模型结果、IC 检测模型结果和必要的兜底结果统一落在整图坐标系中。
- 所有来源的结果最终合并为单一整图结果列表。
- 图像区绘制和文本区输出都只使用整图坐标结果。

## 7. 结果组织与界面展示

### 7.1 最终结果结构

最终结果组织为单一整图结果列表，用于直接驱动 `imagePanel1.UpdateImageAndResult(...)`。

每条结果至少包含：

- `category_name`
- `score`
- `bbox` 或 `rotated_bbox`
- 整图坐标
- `source_model`（可选，用于区分来自元件检测模型、IC 检测模型或兜底结果）

### 7.2 图像区展示规则

- 图像区始终显示原图。
- 可视化对象只使用最终结果，不展示中间滑窗框和中间 ROI。
- 每条结果都绘制边界框和标签。
- 标签格式为：`{category_name} {score:F2}`
- 用户可以在图像区进行滚轮缩放、左键拖拽和右键复位。
- 无结果时仍显示原图，文本区输出“未检测到结果”。

### 7.3 文本区展示规则

文本区显示以下信息：

- 当前图片路径
- 当前元件提取模型路径
- 当前元件检测模型路径
- 当前 IC 检测模型路径
- 当前滑窗参数
- 当前元件外扩参数
- 滑窗数量
- 元件提取模型合并后目标数
- 元件检测模型结果数
- IC 检测模型结果数
- 最终结果数
- 结果明细
- 推理阶段与进度状态
- 错误与异常信息

## 8. 关键业务规则示例

### 8.1 IC 分流示例

元件提取模型某条合并结果为：

- `category_name = IC90`

处理过程为：

- 先按 `元件外扩 = 32` 扩大 ROI
- 解析得到 `base_name = IC`、`angle = 90`
- ROI 逆时针旋转 `90` 度，归一到 `0` 度视角
- 因为 `base_name = IC`，所以该 ROI 走 `IC 检测模型`

若 IC 检测模型输出：

- `IC`
- `引脚`
- `焊点`

则上述结果全部回写到整图坐标，并进入最终结果列表。

### 8.2 普通元件分流示例

元件提取模型某条合并结果为：

- `category_name = 电阻270`

处理过程为：

- 先按 `元件外扩 = 32` 扩大 ROI
- 解析得到 `base_name = 电阻`、`angle = 270`
- ROI 逆时针旋转 `270` 度，归一到 `0` 度视角
- 因为 `base_name` 不是 `IC` 或 `BGA`，所以该 ROI 走 `元件检测模型`

若元件检测模型输出：

- `电阻`
- `焊点`

则上述结果全部回写到整图坐标，并进入最终结果列表。

### 8.3 边界规则示例

- 检测模型无结果：保留当前元件提取结果作为兜底结果
- 元件提取类别角度解析失败：按 `0` 度处理
- 元件外扩后超出图像边界：自动裁剪回有效区域
- 滑窗重叠区域出现重复目标：先映射到整图，再按全图规则合并为一条

## 9. 异常与兜底行为

- 角度解析失败：流程继续执行，当前目标按 `angle = 0` 处理
- 元件外扩后越界：自动裁剪到图像有效区域，不中断流程
- ROI 无效：跳过当前目标并记录日志，不中断整图流程
- 检测模型无结果：保留对应的元件提取结果作为兜底
- 单个 ROI 在局部推理时抛异常：记录异常并继续处理下一个目标
- 模型加载失败：文本区显示异常信息，界面保持可继续操作状态
- 图片解码失败：文本区显示失败信息，不崩溃
- 推理后台线程抛异常：异常回传主线程并显示在文本区，进度条与推理触发控件状态正确复位
- 推理进行中再次触发：提示“当前正在推理”，忽略重复请求
- 释放模型失败：文本区提示异常，但界面不中断

- 选择图片后会自动触发推理，`执行推理` 按钮仍可手动触发
- 推理进度条默认显示 `0%` 且始终可见，推理阶段状态可见
- 推理在后台线程执行，窗口拖拽、最小化、切换时 UI 不明显卡死
- 模型加载固定使用 `device_id = 0`
- 界面中没有 GPU 选择、JSON 输出、一致性测试、测速和测速轮数入口
- 整图滑窗参数默认为 `2560/2560/1024/1024`
- `元件外扩` 默认值为 `32`，且界面位置在 `垂直重叠` 后面
- 滑窗边缘覆盖完整，不漏右边界和下边界
- 元件提取模型结果先抬升整图坐标，再做全图级去重合并
- 合并按完整 `category_name` 生效，不混淆 `IC0` 与 `IC90`、`电阻0` 与 `电阻270`
- `IC`、`BGA` 会路由到 `IC 检测模型`，其他类别会路由到 `元件检测模型`
- 检测结果会先逆旋转回原方向，再回写到整图坐标
- 最终结果为所有分支结果的统一整图列表
- 文本区和图像区能互相印证结果正确性
